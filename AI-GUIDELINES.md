# AI 工具使用规范

本文件为 AI 编码助手（Claude Code、Cursor、Copilot 等）提供项目约束和规范。

## 🚫 禁止操作

以下操作**严格禁止**，除非收到明确的用户指令：

### 文件创建/修改
- ❌ **随意创建新文件** - 只在用户明确要求时创建文件
- ❌ **修改未要求修改的文件** - 即使是为了"改进"
- ❌ **重命名或移动文件** - 除非用户明确要求
- ❌ **删除文件或目录** - 必须用户确认

### 代码生成
- ❌ **生成"可能有用"的代码** - 只生成明确要求的代码
- ❌ **添加"最佳实践"代码** - 未要求不要添加
- ❌ **重构或优化现有代码** - 除非有明确指令

### 文档和配置
- ❌ **修改配置文件** - `.env`, `openclaw.json` 等
- ❌ **更新 README.md** - 除非用户明确要求
- ❌ **创建额外文档** - 超出必需范围的文档

## ✅ 允许的操作

### 只执行用户明确要求的任务
- ✅ 用户说"创建 X 文件" → 创建 X 文件
- ✅ 用户说"修复 X bug" → 修复 X bug
- ✅ 用户说"重构 X 函数" → 重构 X 函数

### 分析和只读操作
- ✅ 读取文件内容进行分析
- ✅ 使用 Glob/Grep 搜索文件
- ✅ 使用 Read 工具查看代码

### 报告和建议
- ✅ 报告发现的问题
- ✅ 提出改进建议（但不自动执行）

## 📁 项目结构约束

### 必须遵守的目录结构

```
harmony-analyse-system/
├── src/mcp/           # MCP Server 源码（只添加新 MCP）
├── skills/             # Skills 定义（只添加新 Skill）
├── plugins/            # 插件代码（已完整，不要修改）
├── scripts/            # 工具脚本（已完整，不要修改）
├── tests/              # 测试文件（可添加新测试）
├── docs/               # 文档（可添加新文档）
├── data/               # 运行时数据（Git 忽略）
└── dist/               # 编译输出（Git 忽略）
```

### 不要创建的文件

- 随机的工具脚本
- 额外的配置文件
- 未要求的功能模块
- "示例"或"演示"文件

## 🛠️ 开发规范

### 新增 MCP Server

1. **创建目录**: `src/mcp/<server-name>/`
2. **必需文件**:
   - `server.ts` - 服务器实现
   - `index.ts` - 入口文件

3. **更新**:
   - `openclaw.config.json` - 添加 MCP server 配置
   - `CLAUDE.md` - 更新 MCP 列表

### 新增 Skill

1. **创建文件**: `skills/<skill-name>.skill`
2. **内容**: 必须包含技能说明、触发条件、MCP 依赖

### 修改现有代码

**必须先询问用户**：
- "我需要修改 X 文件，是否继续？"
- 说明修改原因和范围

## 🔒 关键文件保护

以下文件**需要特别谨慎**，修改前必须确认：

| 文件 | 说明 | 修改要求 |
|------|------|---------|
| `openclaw.config.json` | OpenClaw 配置 | ⚠️ 必须确认 |
| `package.json` | 项目依赖 | ⚠️ 必须确认 |
| `.env.example` | 环境变量模板 | ⚠️ 必须确认 |
| `plugins/openclaw-harmony-security/` | 插件核心 | ⚠️ 已完整实现 |
| `CLAUDE.md` | AI 指南 | ✅ 可更新 |

## 📋 任务执行检查清单

在执行任何写入操作前，确认：

1. [ ] 用户是否明确要求此操作？
2. [ ] 是否在允许的目录/文件范围内？
3. [ ] 是否会影响现有功能？
4. [ ] 是否应该先询问用户？

## 💬 与用户沟通

### 不确定时，这样问：

```
我注意到 [问题]，建议 [解决方案]。
是否继续？
```

### 而不是直接做：

```
❌ "我帮你优化了 X 函数"
❌ "我创建了 Y 工具来提高效率"
❌ "我发现一个问题并修复了它"
```

## 🎯 特殊场景

### 代码审查时
- ✅ 可以指出问题
- ✅ 可以建议改进
- ❌ 不要直接修改代码

### 调试问题时
- ✅ 可以使用 Read/Grep/Bash 分析
- ✅ 可以运行测试
- ❌ 不要随意修改代码

### 用户说"继续"时
- 继续当前明确的任务
- 不要擅自扩展范围
- 完成后报告结果，不要自行开始下一个任务

## ⚠️ 红灯信号

如果用户说：
- "停"、"取消"、"不要"
- "怎么修改了 X？"
- "我没让你做这个"

**立即停止**，解释做了什么，并询问如何处理。

## 📝 典型错误示例

```
❌ "我帮你创建了单元测试框架"
❌ "我注意到你的代码可以优化，帮你重构了"
❌ "我觉得目录结构可以改进，帮你调整了"
❌ "我创建了一个脚本来自动化构建"
```

## ✅ 正确操作示例

```
✅ "收到：创建 X 文件" → 创建 X 文件
✅ "正在分析代码..." → 只分析，不修改
✅ "我发现一个 bug，在 Y 文件第 Z 行" → 只报告，不修复
✅ "需要修改 X 文件吗？" → 询问后再操作
```

## 📞 疑问处理

**当遇到以下情况，先询问用户：**

- 不确定是否应该做某事
- 发现可能的问题但用户没提到
- 看到可以"改进"的地方
- 任务范围不清晰

---

**原则**: 宁可多问，不要擅自行动。用户需要什么，就做什么，不多不少。
